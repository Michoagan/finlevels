# FinLevels Landing & Challenge System — Technical Documentation

## 1. Project overview

FinLevels is a financial behavior and habit-building product. The current application is a landing/quiz/challenge experience that lets users:

1. Take a financial quiz.
2. Receive a financial profile based on three coins:
   - Stability
   - Saving
   - Investing
3. Enter a challenge journey mapped to their weakest/current focus coin.
4. Receive challenge emails every two days through Resend.
5. Open private encrypted challenge links.
6. Complete daily missions and build a streak.
7. View a profile evolution path from their encrypted token.

The current implementation is focused on acquisition, profile collection, personalized challenge delivery, and early retention loops.

## 5. Data architecture

### Supabase tables

#### `waitlist`

Stores users and quiz profile data.

Expected fields:

```sql
id bigint primary key
email text unique not null
source text
profile_name text
stability_level integer
saving_level integer
investing_level integer
coin_priority text[] or jsonb
primary_focus_coin text
created_at timestamptz
```

Current code expects at least:

```txt
id
email
profile_name
stability_level
saving_level
investing_level
primary_focus_coin
```

#### `challenge_email_sends`

Tracks which challenge emails were sent.

This table controls challenge progression.

```sql
create table if not exists public.challenge_email_sends (
  id uuid primary key default gen_random_uuid(),
  user_id bigint not null references public.waitlist(id) on delete cascade,
  path text not null,
  day integer not null,
  sent_at timestamptz not null default now(),
  constraint challenge_email_sends_path_check
    check (path in ('stability', 'saving', 'investing')),
  constraint challenge_email_sends_day_check
    check (day >= 0),
  constraint challenge_email_sends_unique_user_path_day
    unique (user_id, path, day)
);

create index if not exists challenge_email_sends_user_sent_at_idx
on public.challenge_email_sends(user_id, sent_at desc);

create index if not exists challenge_email_sends_user_path_day_idx
on public.challenge_email_sends(user_id, path, day);
```

#### `challenge_completions`

Tracks which challenges the user completed.

This table controls streak and completed UI state.

```sql
create table if not exists public.challenge_completions (
  id uuid primary key default gen_random_uuid(),
  user_id bigint not null references public.waitlist(id) on delete cascade,
  path text not null,
  day integer not null,
  challenge_title text not null,
  completed_at timestamptz not null default now(),
  constraint challenge_completions_path_check
    check (path in ('stability', 'saving', 'investing')),
  constraint challenge_completions_day_check
    check (day >= 0),
  constraint challenge_completions_unique_user_path_day
    unique (user_id, path, day)
);

create index if not exists challenge_completions_user_path_day_idx
on public.challenge_completions(user_id, path, day);
```

### Important data principle

```txt
challenge_email_sends = what mission we sent
challenge_completions = what mission the user completed
```

These must stay separate.

Email sends control progression. Completions control streak.

## 6. Token architecture

Challenge and profile links use encrypted tokens generated by `lib/challenge-token.ts`.

Token payload:

```ts
{
  userId: number;
  path: "stability" | "saving" | "investing";
  day: number;
}
```

Encryption:

- AES-256-GCM
- Secret: `CHALLENGE_TOKEN_SECRET`
- Token format starts with version prefix:

```txt
v1.iv.encryptedPayload.authTag
```

### Why tokens are used

Old challenge URLs exposed details:

```txt
/challenges?path=stability&day=0&user=email@example.com
```

Current URLs are private:

```txt
/challenges/v1.encrypted-token
/profile/v1.encrypted-token
```

They do not expose:

- Email
- User ID
- Challenge path
- Challenge day

## 7. Challenge content source of truth

The source of truth for challenge content is:

```txt
lib/challenge-email-templates.ts
```

It exports:

```ts
challengeEmailTemplates
getChallengeEmailTemplate(path, day)
```

Each challenge template includes:

```ts
{
  path,
  day,
  title,
  subject,
  previewText,
  greeting,
  intro,
  previousChallengeCta,
  lesson,
  challenge,
  currentChallengeCta,
  outro,
  signature,
}
```

The challenge page now reads the displayed mission from this file. The Resend cron also reads email content from this file.

That means changing challenge text in `challenge-email-templates.ts` updates:

- Email copy
- Web challenge page mission copy

### Note about `app/challenges/challenge-content.ts`

This file still exists from an earlier local challenge UI system. The active challenge page is now wired to `challenge-email-templates.ts`. If no code imports `challenge-content.ts` in the future, it can eventually be removed or repurposed.

## 8.2 Manual challenge lookup flow

Route:

```txt
/challenges
```

Flow:

```txt
User enters email
  ↓
POST /api/challenges/progress
  ↓
Server checks waitlist using service role key
  ↓
If no quiz profile: returns null
  ↓
Client redirects to /quiz
```

If profile exists:

```txt
Server reads latest challenge_email_sends row
  ↓
Determines next day from email-send progression
  ↓
Clamps max day using challengeEmailTemplates[path].length
  ↓
Generates encrypted token
  ↓
Client redirects to /challenges/[token]
```

## 8.3 Email challenge flow

Route created by email:

```txt
/challenges/[token]
```

Flow:

```txt
User clicks email CTA
  ↓
/challenges/[token]
  ↓
Server decrypts token
  ↓
Server reads challenge_completions for current user/path
  ↓
Server calculates streak and whether current day is completed
  ↓
Client renders challenge from challengeEmailTemplates[path][day]
```

If the challenge day has already been completed:

- Validation buttons are hidden
- Completed state is shown
- Post-completion CTAs are shown

If not completed:

- User sees mission
- User must enter proof text in textarea
- User can click `Yes, completed`

Current proof status:

```txt
UI-only. Not yet saved to Supabase.
```

## 8.5 Streak logic

Implemented in:

```txt
lib/challenge-streak.ts
```

Rule:

```txt
Streak = consecutive completed days starting from day 0.
```

Examples:

```txt
[]        -> 0
[0]       -> 1
[0, 1]    -> 2
[0, 2]    -> 1
[1, 2]    -> 0
[0,1,2,4] -> 3
```

This means skipping a day breaks streak progress even if later days are completed.

## 9. Resend email automation

### Cron route

```txt
/api/cron/send-challenge-emails
```

Implemented in:

```txt
app/api/cron/send-challenge-emails/route.ts
```

### Vercel schedule

Configured in:

```txt
vercel.json
```

```json
{
  "crons": [
    {
      "path": "/api/cron/send-challenge-emails",
      "schedule": "0 9 */2 * *"
    }
  ]
}
```

This means:

```txt
Every 2 days at 09:00 UTC
```

### Cron flow

```txt
Vercel Cron calls /api/cron/send-challenge-emails
  ↓
Route checks CRON_SECRET if configured
  ↓
Fetches quiz users from waitlist
  ↓
For each user:
  - validate email and primary_focus_coin
  - read latest challenge_email_sends day
  - nextDay = lastSentDay + 1, or 0 if none sent
  - get template from challengeEmailTemplates[path]
  - if no next template, skip user as track completed
  - check previous sent days vs completed days
  - include missed previous challenge CTA if applicable
  - generate encrypted current challenge token
  - send email through Resend HTTP API
  - upsert challenge_email_sends row
```

### Email progression rule

```txt
Email sends control progression.
Challenge completions do not block the next email.
```

Example:

```txt
Day 0 sent, not completed
2 days later Day 1 still sends
Email can include reminder link to Day 0
```

### Resend API call

The app calls Resend directly with `fetch`:

```txt
POST https://api.resend.com/emails
Authorization: Bearer RESEND_API_KEY
```

Payload includes:

```ts
{
  from,
  to,
  subject,
  html,
}
```

### Cron response shape

Manual trigger returns summary:

```json
{
  "sent": 1,
  "skipped": 0,
  "failed": 0,
  "results": [
    {
      "userId": 13,
      "email": "test@example.com",
      "path": "stability",
      "day": 0,
      "status": "sent"
    }
  ]
}
```

```json

  "email": "user@example.com"
```

- Normalizes email
- Looks up `waitlist`

```json
{
  "progress": {

  }
```

```json

  "progress": null
```

```json
  "success": true,
```

## 14. Known limitations/current implementation notes

### Proof textarea is UI-only

Users must enter proof before completion, but earlier this proof text was only stored client-side. The current update stores a user-provided short proof on the `challenge_completions` table.

Recommended / applied update:

- Add `proof` column (nullable text) to `challenge_completions`
- Update `/api/challenges/complete` request body to accept `proof`
- Persist `proof` alongside the completion record

Example SQL to add the column (migration-friendly):

```sql
ALTER TABLE IF EXISTS public.challenge_completions
  ADD COLUMN IF NOT EXISTS proof text;

COMMENT ON COLUMN public.challenge_completions.proof IS
  'Optional user-provided proof (short text) for the completed challenge';
```

### Cron route sends to all quiz users

The cron currently fetches all users with `profile_name` and `primary_focus_coin`.

Recommended future improvements:

- Add unsubscribe/preferences table
- Add `challenge_emails_enabled boolean`
- Add `last_email_error` logging
- Add dry-run mode
- Add test-only query param gated by secret

### No unsubscribe handling yet

Production marketing/transactional email should support unsubscribe behavior where appropriate.

### Email HTML is inline and basic

The email renderer is currently a simple function in the cron route. If email design grows, consider moving it to:

```txt
lib/challenge-email-renderer.ts
```

### Profile guidance duplicated

Profile guidance currently exists in `quiz/page.tsx` and `profile/[token]/page.tsx`.

Recommended future cleanup:

```txt
lib/profile-guidance.ts
```

Then import it in both places.

## 16. Mental model summary

```txt
Quiz creates the profile.
Profile chooses the challenge path.
Email sends move the user through days.
Challenge completions build streak.
Templates power both email and web mission content.
Encrypted tokens keep links private.
Cron automates delivery every two days.
```

Most important rule:

```txt
Do not use challenge completions to decide the next email.
Use challenge_email_sends.
```
